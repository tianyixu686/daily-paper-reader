## 摘要
大型语言模型及基于LLM的智能体广泛用作个人聊天助手、企业副驾驶和自主工作流智能体。在这些应用中，记忆（即对长上下文和多次交互中积累的信息进行保留、访问和推理的能力）在决定智能体可靠性方面起着关键作用。我们提出RECON（带混淆叙事的扩展上下文推理），一个评估长上下文组合推理的基准。RECON涵盖三个领域（刑事、医疗和金融）的24个案卷，每个案卷长度在5万到10万词之间，测试智能体在六项记忆密集型任务上的表现：重构多跳证据链、传播级联失效、解决源冲突、反事实推理、满足时间约束以及时间事实检索。近期记忆基准评估智能体是否能检索分散的事实或检测事实是否发生变化，而RECON评估变化发生后会发生什么——智能体能否追踪哪些下游结论受影响、哪些通过独立支持而存续、以及替代时间线会如何展开。我们的评估揭示了当前架构的显著局限性：即使最强的非Oracle系统也仅达到22.4%的准确率，其中检索和推理各自构成了挑战。

## Abstract
Large language models and LLM-based agents are widely used as personal chat assistants, enterprise copilots, and autonomous workflow agents. In all these applications, memory (the ability to retain, access, and reason over information accumulated over long contexts and multiple interactions) plays a crucial role in determining the reliability of any agent. We introduce RECON (Reasoning over Extended Contexts with Obfuscated Narratives), a benchmark for evaluating compositional reasoning over long contexts. RECON spans 24 case files across three domains (criminal, medical, and financial), each ranging from 50k to 100k tokens, and tests agents on six memory intensive tasks: reconstructing multi-hop evidence chains, propagating cascading invalidations, resolving source conflicts, counterfactual reasoning, satisfying temporal constraints, and temporal fact retrieval. Recent memory benchmarks evaluate whether agents can retrieve scattered facts or detect if a fact has changed whereas RECON evaluates what happens after the change, whether agents can trace which downstream conclusions are affected, which survive through independent support, and how alternative timelines would have unfolded. Our evaluation reveals substantial limitations across current architectures: even the strongest non-Oracle system reaches only 22.4% Accuracy, with retrieval and reasoning each surfacing as challenges.

- **核心问题**：智能体能否在长上下文（5万-10万 token）中，对**演化而非静态**的证据进行多步推理，包括追踪依赖关系、解决源冲突、处理级联失效和反事实情景？
- **整体含义**：记忆不仅是事实的存储，更是**推导历史的追踪**。论文指出现有基准将记忆建模为“状态机”是不够的，需要建模为“来源图”（provenance graph），结论携带推导历史，失效事件通过显式依赖传播。

## 2. 论文提出的方法论

### 2.1 核心思想
- 构建一个**确定性生成管道**，每个案卷的内在结构（证据链、失效事件、冲突、反事实依赖、平行时间流）完全由程序生成，**不依赖 LLM 定义因果或答案**。LLM 仅用于将结构化骨架转化为自然语言叙述（受约束的表面实现）。
- 智能体只能看到叙述后的案卷文件，必须从中恢复出证据的依赖图、执行六类记忆密集型任务。

### 2.2 关键技术细节
- **生成管道六阶段**：
  1. **属性条件蓝图合成**：基于种子、域目录、配额，通过产生式规则（类似文法）生成结构化蓝图。每条规则有前置条件（requires）和后置（establishes），确保依赖图可追溯。
  2. **骨架扩展与时间强制**：将蓝图展开为带有时间戳的事件、证据条目、因果依赖、失效、冲突、反事实依赖、平行时间流（监控、访问、交易、临床、审计）。强制**字典序时间单调性**约束：每个节点有 (day, time)，因果边要求后序节点时间不低于前序。
  3. **证据 DAG 构建**：从骨架中直接导出有向无环图（节点为事件/证据/结论，边为因果、修改、失效、冲突解决关系）。该图是权威表示。
  4. **确定性任务合成**：从 DAG 中算法生成 6 类问题。例如：级联任务选择失效节点，通过可达性计算哪些结论坍塌/存活；反事实任务改变时间戳重新计算可达性。
  5. **语言实现**：LLM 将骨架条目转化为调查风格散文，但骨架字段（时间、来源、实体、证据 ID）不可修改。后验证确保忠实。
  6. **六阶段验证**：正则/模式检查、结构验证（DAG 无环、时间单调、引用完整）、来源-接地验证、LLM 语义审计、鲁棒性审计（匿名化、时间戳归一化）、人类验证（200 个问题的层化样本，Fleiss’ κ=0.69）。

### 2.3 任务类型（6类）
| 任务 | 描述 |
|------|------|
| **链重构 (Chain Reconstruction)** | 定位并因果排序5-15个证据跳，相邻跳词汇不相似且有大量中间条目 |
| **级联传播 (Cascade Propagation)** | 给定正式失效（样本污染、证词撤回等），判断哪些结论失去支持，哪些通过独立证据存活 |
| **源冲突解决 (Source Conflict Resolution)** | 通过定位独立佐证证据（未标注、距离远）来调解矛盾叙述 |
| **反事实推理 (Counterfactual Reasoning)** | 如果早期发现发生在不同时间或未发生，下游事件如何变化；保持实际和假设时间线区分 |
| **时间约束满足 (Temporal Constraint Satisfaction)** | 跨平行时间流（监控、访问、交易、通信、传感器、临床、审计）交叉参考满足约束 |
| **时间事实检索 (Temporal Fact Retrieval)** | 控制类任务：事件排序、时间点状态、流匹配、有明确证据的排除 |

## 6. 论文的主要结论与发现

1. **RECON 极难**：最佳非 Oracle 系统（Gemini-2.5-Pro）准确率仅 **22.4%**；Oracle（获得完整结构化 DAG）也只有 **54.6%**。
2. **检索和推理都是瓶颈**：
   - 在“完整覆盖率”问题（支持证据全部被检索到）中，仍有约 **80% 的问题被回答错误**（长上下文家族平均准确率 14.8% vs Oracle 54.2%，差距 39.4 pp 反映推理瓶颈）。
   - 源冲突、级联传播、链重构的 Oracle–LLM 差距最大（+59 至 +64 pp），表明检索是主要瓶颈；反事实和时间约束差距小（+13-18 pp），因为 Oracle 本身也受限。
3. **不同架构各有所长**：
   - **长上下文**在四项多源推理任务（链重构、源冲突、反事实、时间约束）领先。
   - **RAG** 在时间事实检索上得分最高（最佳变体 hybrid+rerank 0.443 vs 最佳 LLM 0.490）。
   - **记忆系统**在级联传播上表现卓越（Supermemory 0.708，远高于所有长上下文和 RAG）。
   - 没有任何系统在所有任务上表现好。
4. **链长度严重影响性能**：Oracle 准确率从链长 3-4 的 100% 单调下降到链长 10 的 50%；非 Oracle 系统在所有链长度上基本接近零。
5. **反事实推理是硬天花板**：最佳非 Oracle 反事实分数仅 0.305，Oracle 0.483，表明推理（而非检索）是瓶颈。
6. **记忆系统 token 效率高**：长上下文每个问题约 65k-70k token，RAG 约 7k-9k，记忆系统约 3k-8k，但压缩导致链重构和源冲突性能下降（丢失了事实间边的信息）。
7. **叙述者家族偏差不显著**：同家族 vs 跨家族准确率差异统计上不显著 (p>0.3)。

## 7. 优点

- **确定性生成**：所有 ground truth 由程序推导，不依赖 LLM，确保正确性和复现性，避免 LLM 评价 LLM 的循环。
- **任务设计全面且现实**：涵盖级联失效、源冲突、反事实推理等真实场景，超越静态事实检索。
- **人类验证**：在 200 个问题上达到 κ=0.69 的实质一致性，定性评价（清晰度、证据充分性、叙述忠实性）均在 90% 左右。
- **污染过滤与鲁棒性检查**：排除闭卷可猜问题，并测试匿名化/时间戳归一化对答案稳定性的影响，减少潜在捷径。
- **开放代码与基准**：提供生成管道、基准、评估框架，促进后续研究。
- **跨架构比较**：系统比较了长上下文、RAG、记忆系统三种范式，揭示不同架构的优势和短板。

## 8. 不足与局限

- **合成数据**：叙述由 LLM 生成，尽管严格受 skeleton 约束，但训练于类似合成文本的模型可能有风格优势。论文通过使用独自的实体和证据结构部分缓解，但无法完全排除。
- **模式泄漏风险**：了解内部骨架模式的可提取管道可能利用答案格式而非真正推理。
- **人类基线不公平比较**：人类评估者获得了问题相关的证据包（子图+叙述摘录），而 LLM 求解器面对完整 100k token 案卷或自检索上下文。因此人类准确率（63.0%）应视为与 Oracle 求解器互补的上界，而非与长上下文系统的直接比较。
- **计算资源未报告**：论文未提及训练或评估所需的具体 GPU 类型、数量或时间，不利于可重复性比较。
- **任务分布权重固定**：按固定矩阵（表 4）分配问题，可能导致某些类型过少或过多，但表中显示均衡。
- **免费形式评分依赖 LLM**：尽管使用两个不同家族的 LLM 法官来减少自我偏好，但 LLM 作为评估者仍有内在不确定性。
