## 摘要
推理上下文长度与RL后训练之间的差距日益扩大：推理系统正在接近百万token的上下文，而后训练工作负载通常仍保持在256K token或以下，并依赖部署时的长度泛化。这种差距对于AI智能体尤其重要，其观察结果、工具输出、文档和先前决策会在长轨迹中累积。LongStraw是一个面向架构的、在固定GPU预算下进行百万token RL后训练的执行栈，通过分组相对策略优化（GRPO）实例化。它在无需自动求导的情况下评估共享提示，仅保留后续token所需的模型特定状态，并逐一重放短响应分支，以减少实时训练图，代价是增加额外重放时间。我们将其实现用于混合循环和全注意力Qwen3.6-27B以及压缩注意力混合专家GLM-5.2。在八块H20 GPU上，LongStraw完成分组Qwen评分和响应反向传播，覆盖2.1M位置，组大小分别为2和8；增加组大小仅增加0.21 GB的峰值分配内存，而单独的压力测试达到4.46M位置。在32块H20 GPU上，我们验证了针对GLM-5.2全部78层的2.1M token提示的端到端LongStraw执行路径。这些实验确立了执行能力而非完整的训练正确性，因为捕获的提示状态是分离的，且一些分布式前向和梯度组合路径仍不完整。

## Abstract
A growing gap separates inference context lengths from RL post-training: inference systems are approaching million-token contexts, while post-training workloads often remain at 256K tokens or below and rely on length generalization at deployment. The gap is especially important for AI agents, whose observations, tool outputs, documents, and prior decisions accumulate over long trajectories. LongStraw is an architecture-aware execution stack for million-token RL post-training under a fixed GPU budget, instantiated with Group Relative Policy Optimization (GRPO). It evaluates the shared prompt without autograd, retains only model-specific state needed by later tokens, and replays short response branches one at a time, reducing the live training graph at the cost of additional replay time. We implement it for the hybrid recurrent and full-attention Qwen3.6-27B and the compressed-attention mixture-of-experts GLM-5.2. On eight H20 GPUs, LongStraw completes grouped Qwen scoring and response backward at 2.1M positions for groups of 2 and 8; increasing the group size adds only 0.21 GB of peak allocated memory, while a separate stress test reaches 4.46M positions. On 32 H20 GPUs, we validate the end-to-end LongStraw execution path for a 2.1M-token prompt across all 78 layers of GLM-5.2. These experiments establish execution capacity rather than complete training correctness because the captured prompt state is detached and some distributed forward and gradient composition paths remain incomplete.

# 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：推理阶段的上下文长度已接近百万token，但强化学习（RL）后训练（post-training）通常限制在256K token或以下，大量工作依赖长度泛化（length generalization）在推理时处理更长序列。这一差距对AI代理尤其严重：代理的决策需要累积观察、工具输出、文档和先前行动，构成长期轨迹；而短上下文后训练导致梯度无法有效传播到早期信息。
- **整体含义**：LongStraw旨在固定GPU预算下（不增加加速器数量），实现百万token量级的RL后训练执行，从而突破长度泛化的瓶颈，降低长上下文训练对大规模GPU集群的依赖，使更多研究者和团队能够探索长上下文代理训练。

## 2. 方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- **分离共享提示与响应分支**：一个分组更新中，所有响应共享同一个提示（prompt）。LongStraw将提示评估与响应分支的后向传播解耦——提示只计算一次，**不保留自动求导图**，仅存储后续响应所需的架构特定状态（如注意力KV页、循环状态）；然后逐一重放短响应分支（response branch），仅对当前响应分支保持自动求导图，释放后处理下一个。

### 关键技术细节
- **提示捕捉（Prompt Capture）**：使用`torch.no_grad()`评估所有提示token，释放临时激活（注意力scratch、FFN中间结果、MoE路由等），仅保留持久性状态。对于Qwen（混合循环+全注意力）保留GDN（Gated Delta Network）循环状态和全注意力层的键值页；对于GLM（压缩注意力+MoE）保留MLA（Multi-head Latent Attention）潜变量页和DSA（DeepSeek Sparse Attention）索引键页，并移至CPU。
- **响应重放（Response Replay）**：逐个响应分支执行：首先运行旧策略/参考策略评分（无梯度），然后重建当前策略响应图（带梯度），后向传播，立即释放该分支的激活。所有分支的梯度累加后，调用一次优化器。
- **分组相对策略优化（GRPO）**：使用广义GRPO目标（公式1-2），采用组内奖励归一化、裁剪比率和参考KL惩罚。LongStraw关注执行图的边界，而非修改优化器本身。
- **物理所有权与紧凑存储**：上下文并行（CP）下，逻辑页面必须占有紧凑物理分配（复制而非视图），否则父分配无法释放。Qwen使用8路CP，每个rank拥有4,080个页面（每页64 token）；GLM使用32路CP的zigzag布局，每个rank拥有1,024页（65,536 token）。
- **全层检查点（Whole-Layer Checkpointing）**：对于GLM的MoE响应图，检查点边界设为完整解码器层（而非仅注意力），以避免保留MoE路由和专家中间结果。重放时逐层从CPU加载状态到GPU。
- **并行布局**：Qwen仅用CP8（8 H20）；GLM使用TP1/CP32/EP32/ETP1/PP1（32 H20），CP和EP分布在相同rank上但负责不同对象（上下文 vs 专家）。
- **梯度缺失项**：由于提示状态分离，响应后向传播只计算 `∂ℓ/∂θ` 的第一项（显式响应计算），而忽略 `∂ℓ/∂z_P * ∂z_P/∂θ`（提示状态的参数导数），因此不等价于完整序列梯度。

### 公式或算法流程
- **公式1-2**：GRPO目标包含裁剪策略项和KL惩罚，作者重点在于如何执行这些条件对数概率。
- **公式3**：完整梯度分解，指出LongStraw只计算第一项。
- **执行流程**：四阶段——①提示捕捉（无梯度）；②预先评分（旧/参考策略冻结）；③逐个响应分支重放（旧评分→参考评分→策略前向→后向→释放图）；④优化器调用（累加梯度后一次步骤）。

- **输入**：合成随机token（Qwen: 2,088,960提示 + 8,192响应输入；GLM: 2,097,152提示 + 少量响应token）。奖励为确定性函数（GLM使用 `[0,1]` 生成优势 `[-1,1]`）。

## 6. 论文的主要结论与发现

- **固定GPU预算下，长上下文GRPO执行是可行的，关键在于张量生命周期和物理所有权管理**——通过分离提示图并串行重放响应，激活规模由响应长度主导而非提示长度。
- **Qwen**：在8块H20上，2.1M上下文、G=2和8均完成执行，峰值为97.5/97.7 GB；组大小从2增至8，峰值内存仅增加0.21 GB（0.213%），时间增加约1,586秒。4.25M上下文（prefix-frozen）完成8次G=8优化步骤（64次成员重放），峰值83.9 GB。
- **GLM**：在32块H20上，2.1M提示+两个响应分支的两个78层后向传播和优化器调用完成执行。
- **关键发现**：密集模型（Qwen）和混合专家模型（GLM）的主要瓶颈不同——密集模型线性存储开销来自KV页面，MoE模型则来自专家路由和激活膨胀。上下文并行和专家并行是正交的，不能互相替换。
- **分布式梯度组成的缺失**：提示状态梯度被忽略；Qwen的K/V投影适配器梯度未被同步；GLM跳过Megatron的梯度最终化步骤。因此，收据属于“执行容量”级别，而非“分布式更新一致性”或“全梯度等价性”。

## 7. 优点

- **方法创新**：提出基于“捕捉共享提示+串行重放响应”的固定预算执行范式，系统性地将长上下文训练从扩展设备转移到优化状态生命周期。
- **架构泛化性**：对两种截然不同的模型架构（密集混合注意力 vs 压缩注意力+MoE）均实现了可行性，方案具有架构适配性。
- **诚实透明**：明确区分了四个证据等级，并详细记录了当前实现中缺失的分布式操作和未完成的验证，为后续工作提供了清晰的路线图。
- **渐进式构建**：GLM路径从32K开始，逐步添加依赖（前缀容量→单层重放→全层架构→并行所有权→组执行），证明了每一步的约束和解决方案。
- **开源贡献**：代码已公布在GitHub，增强可复现性。
- **资源效率**：在相对少量的GPU（8/32块H20）上实现了百万级上下文后训练执行，降低了硬件门槛。

## 8. 不足与局限

- **分布式梯度组成不完整**：Qwen缺少K/V投影适配器的跨秩归约；GLM跳过`finalize_model_grads`，导致CP复制适配器梯度未归约。优化器调用基于局部梯度，参数更新可能不一致。
- **提示状态梯度被分离**：即使修复分布式梯度，仍忽略 `∂z_P/∂θ` 项，不等同于完整序列梯度。未进行短上下文下的全梯度等价性比对。
- **执行仅为一次探测**：所有输入为合成数据，奖励为简单确定性函数；无在线抽样、环境交互、多步更新循环；无后训练效果评估（如任务准确率）。
- **GLM的DSA操作仅局部**：每个rank仅从自己的65,536 token分片中选取top-2048，而非全局2M context，因此前向算子不等于模型定义的多头稀疏注意力。当前代码树已实现全局DSA，但未在2M上重新运行验证。
- **内存测量不完整**：仅报告CUDA分配器峰值（max_memory_allocated），不包括预留内存、进程内存；GLM仅报告前缀捕捉窗口峰值，未记录完整事务峰值。Qwen的4.25M clean path OOM未经准确allocator记录。
- **无对比实验**：未在同一硬件上运行其他长上下文训练系统（如Ring Attention等），无法进行效率/内存对比。
- **复现性限制**：GLM最终收据未绑定学习率、token校验和、软件版本等，依赖早期文档而非规范收据。
- **有效性有待验证**：由于未进行完整正确性验证和策略评估，该工作属于系统工程可行性报告，而非可落地的训练方案。按作者自己提出的路线图，下一步需先修复分布式梯度归约，再进行全梯度等价性验证和重复训练循环。
